# code/bspc/aas_edgemelting.h — Enhanced Analysis

## Architectural Role

This header module declares the public interface for a **post-processing pass within the BSPC (BSP→AAS compiler) offline geometry pipeline**. The edge-melting pass consumes area face windings generated by prior phases (`aas_create.c`, `aas_facemerging.c`) and outputs simplified, colinear-edge-free geometry that reduces redundancy before final AAS serialization. Within the broader engine, botlib's runtime AAS system loads the compiled `.aas` files and performs navigation queries—the quality and compactness of edge-melted geometry directly impacts both bot pathfinding fidelity and runtime memory footprint.

## Design Patterns & Rationale

### Pipeline Architecture Pattern
BSPC implements a **linear feed-forward compilation pipeline**: each pass consumes geometry from the prior stage, transforms it, and hands off to the next. Edge-melting is a **cleanupOptimization pass**, positioned after face merging but before topological clustering—this ordering allows:
- Face merging to coalesce same-plane faces (horizontal scope)
- Edge melting to eliminate redundant colinear vertices within each merged face (vertical scope)
- Clustering to then operate on simplified, canonical face geometry

### Batch, No-Argument Interface
The signature `void AAS_MeltAreaFaceWindings(void)` implies:
- **Global state encapsulation**: all working geometry lives in module-static or well-known global structures (e.g., `aasworld`), avoiding parameter-passing overhead for a large dataset
- **Single-pass semantics**: one invocation processes the entire loaded map; idempotent within a compilation run
- **Monolithic design era**: typical of late-1990s offline tools before parameterization became standard (cf. q3map's similar monolithic architecture)

## Data Flow Through This File

**Input:** Area/face/winding data already loaded and partially coalesced by `AAS_Create` and `AAS_MergeAreaFaces`
→ **Transform:** For each area face, detect and remove colinear (redundant collinear) vertex sequences; compact winding vertex counts
→ **Output:** Simplified face windings fed directly to downstream `AAS_InitClustering` and reachability passes

Edge melting is a **data reduction** operation—vertices are culled, not re-emitted or re-indexed; no new structures are created.

## Learning Notes

### Engine-Era Idioms
This file exemplifies **early-2000s offline tool design**:
- Single global state context (vs. modern factory/builder patterns)
- Bare function calls (no vtable dispatch, no struct methods)
- No return codes or error propagation (failures likely call `Error()` directly, as seen in other BSPC modules)
- Header declares implementation boundary; no inline expansion

### Geometric Simplification Strategy
Edge melting is a **vertex-reduction optimization**, different from modern mesh decimation:
- **Colinear removal**: detects three consecutive vertices on the same line, removes the middle
- **Intent**: reduce polygon complexity for faster queries at runtime without altering topology or bot traversability
- **Contrast with botlib's `AAS_Optimize`**: that runtime phase strips entire non-essential face types (ladders, etc.); BSPC edge-melting preserves face structure but compacts vertices

### Connection to AAS Subsystem
The AAS (Area Awareness System) is the core pathfinding backbone for bots. Three key phases:
1. **Compile time (BSPC):** Map → areas → faces → edges → reach/cluster
2. **Load time (botlib):** `.aas` file → `aasworld` singleton; `AAS_Optimize` compaction
3. **Runtime (botlib):** Route queries, entity linking, movement prediction

Edge melting sits at the boundary between phases 1 and 2, reducing serialized file size and load-time memory usage.

## Potential Issues

**None clearly inferable from this header alone.** However, design observations:
- **No validation exposure**: if `AAS_MeltAreaFaceWindings` silently corrupts winding integrity (e.g., creates degenerate polygons), the header provides no feedback; downstream code must detect or tolerate such errors
- **Global state mutation**: any concurrent access to `aasworld` during melting would be catastrophic; single-threaded BSPC assumption is implicit but not documented here
