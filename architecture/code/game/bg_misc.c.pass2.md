# code/game/bg_misc.c — Enhanced Analysis

## Architectural Role

This file implements the **shared gameplay data registry and deterministic physics infrastructure** for Quake III's client/server prediction system. It is compiled identically into both the **game VM** (server) and **cgame VM** (client) to ensure that player movement prediction, item eligibility, and trajectory evaluation produce byte-identical results on both sides. As the sole keeper of `bg_itemlist[]`, it serves as the canonical item taxonomy referenced at spawn time, pickup time, and UI rendering time across all engine subsystems.

## Design Patterns & Rationale

**1. Deterministic Duplication (Twin Compilation Pattern)**
This file is compiled into both game and cgame to guarantee byte-identical physics and item logic on client and server. This is a deliberate architectural constraint: prediction divergence between client pmove and server pmove is catastrophic for gameplay feel. By compiling `bg_pmove.c` and `bg_misc.c` identically into both VMs, the engine avoids the complexity of maintaining two separate physics implementations. The cost is code duplication, but the benefit is **100% deterministic prediction** — a hard requirement for a competitive arena shooter.

**2. Item Registry as Configuration Database**
Rather than scattering item properties across multiple files or loading from external data files, `bg_itemlist[]` is a **compile-time constant array**. This design choice (vs. runtime data tables):
- Eliminates filesystem dependencies for core gameplay
- Ensures items are precached at VM load time
- Allows the compiler to optimize item lookups (linker can even relocate the table)
- Simplifies network transmission (items are identified by index, not name)

**3. Trajectory Type Dispatch (Virtual Dispatch via Switch)**
`BG_EvaluateTrajectory` and `BG_EvaluateTrajectoryDelta` implement a switch-based dispatch for trajectory types (`TR_STATIONARY`, `TR_INTERPOLATE`, `TR_LINEAR`, `TR_LINEAR_STOP`, `TR_SINE`, `TR_GRAVITY`). This is simpler and more efficient than vtable dispatch and allows the compiler to inline hot paths. The cost is that adding a new trajectory type requires a single central edit here, not scattered modifications.

**4. Event Ring Buffer with Sequence Numbers**
`BG_AddPredictableEventToPlayerstate` maintains a two-slot ring buffer (`ps->events[2]`) with a monotonic sequence counter (`ps->eventSequence`). This design:
- Avoids heap allocation for events in the VM sandbox
- Allows cgame to detect missed events (sequence gap = network loss)
- Pairs naturally with delta-compression (delta encoder can skip unchanged events)
- Limits to 2 simultaneous events per frame (conservative; prevents unbounded event generation)

**5. Asymmetric Pickup Proximity (Player AABB vs. Item Point)**
`BG_PlayerTouchesItem` uses an asymmetric bounding box: player extent is full-sized, but item is treated as a point. This is a deliberate gameplay choice: items don't "reach out" to grab you; you must touch them. The function intentionally ignores crouch height difference, which may seem odd but reflects the design intent that ducking doesn't prevent pickups.

## Data Flow Through This File

### Item Spawning Flow (Server → Network → Client)
```
g_spawn (server) 
  → g_items.c spawn_item_funcs (entity_t*) 
    → BG_FindItem(classname) → lookup bg_itemlist
    → entity spawned with itemindex = bg_itemlist index
  
sv_snapshot.c encode_entitystate
  → copy entityindex into snapshot
  
cg_ents.c process_snapshot
  → CG_SnapshotEntityNumToTag (reverse lookup using entityindex)
  → get &bg_itemlist[index]
  → render model/icon based on bg_itemlist[index]
```

### Player State → Network → Client Prediction
```
server frame:
  g_active.c: run pmove on playerState_t (player origin, velocity, etc.)
  g_active.c: BG_PlayerStateToEntityState(playerState_t → entityState_t)
  sv_snapshot.c: delta-encode entityState_t into outgoing snapshot

client frame:
  cg_snapshot.c: decode snapshot → playerState_t
  cg_predict.c: run pmove identically (BG_EvaluateTrajectory for projectiles)
  cg_predict.c: extrapolate forward 50ms via BG_PlayerStateToEntityStateExtraPolate
  cgame renderer: interpolate between snapshots using TR_INTERPOLATE trajectory
```

### Trajectory Evaluation (Physics Simulation)
```
pmove (bg_pmove.c):
  ← BG_EvaluateTrajectory(groundTrace.gravity → accel[2] = -DEFAULT_GRAVITY)
  ← BG_EvaluateTrajectoryDelta (for integration step)
  ← BG_TouchJumpPad (applies velocity + queues EV_JUMP_PAD event)

server snapshot gen:
  ← BG_PlayerStateToEntityState (sets trajectory type = TR_INTERPOLATE)

client rendering:
  ← BG_EvaluateTrajectory (remote player position at render time)
```

## Learning Notes

**Idiomatic Patterns**
- **Shared VM code**: Both game and cgame include `bg_*.c` files to guarantee determinism. Modern engines (e.g., Unreal, Godot) use **unified physics engines** instead, but Quake III's VM sandbox required this duplication.
- **Delta compression + events**: Rather than re-sending full state, Quake III sends deltas. Events provide a separate channel for discrete, unpredictable actions (weapon fire, jumps).
- **Trajectory parametrization**: Using `trajectory_t` (position + velocity + time) to decouple physics simulation from rendering. This allows extrapolation and smooth interpolation without re-running full physics on the client.
- **Item registry immutability**: Unlike modern games with dynamic item properties, Q3A bakes item stats into a compile-time array. This simplifies network sync and prevents cheating via item manipulation.

**Connections to Engine Concepts**
- **Prediction** – This file is the **heartbeat of Q3's prediction system**. Every function exists to support the core loop: client predicts, server authorizes, divergence decays.
- **Event System** – `BG_AddPredictableEventToPlayerstate` is the bridge between deterministic physics (which doesn't produce events) and discrete gameplay (which does). Events allow cgame to react to server-verified actions.
- **State Compression** – `BG_PlayerStateToEntityState` is the **encoder**: full state → network state. The decoder is in `cg_snapshot.c`. This asymmetry reflects the fact that servers need full state; clients only need interpolation-friendly subsets.

## Potential Issues

1. **Implicit Gametype Dependencies in `BG_CanItemBeGrabbed`**: The function's armor/health caps are conditioned on MISSIONPACK and `PW_GUARD` powerup state. If a new gametype or powerup is added without updating this function, item behavior will diverge between game and cgame. No guard against this.

2. **Hardcoded `DEFAULT_GRAVITY` (800) in `BG_EvaluateTrajectory`**: Gravity is burned in; local gravity per-area is noted as "unimplemented" in a comment. If a map designer wanted variable gravity (water zones, low-g moon maps), this would silently fail.

3. **Event Sequence Overflow**: `ps->eventSequence` is a 32-bit int. It will overflow, but the code handles this naturally via unsigned arithmetic (wraparound). However, if a client is disconnected and reconnected mid-frame, sequence continuity is broken — cgame may drop or duplicate events. (This is probably acceptable given frame-rate scales.)

4. **Ammo Cap Hardcoded to 200**: No enum constant, just the number 200 scattered in `BG_CanItemBeGrabbed`. Changing ammo cap requires a code edit + recompile of both game and cgame VMs — no gameplay console cvar.
