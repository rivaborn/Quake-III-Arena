# lcc/cpp/macro.c — Enhanced Analysis

## Architectural Role

This file implements the C preprocessor macro expansion engine for the LCC compiler's preprocessor. As part of `lcc/cpp/`, it handles `#define` directives, macro invocations, and both standard operators (`#`, `##`) and built-in macros (`__LINE__`, `__FILE__`, `__DATE__`, `__TIME__`). The preprocessor is invoked during QVM bytecode compilation, so this module is part of the offline tool chain—not the runtime engine—but is critical for normalizing source code that will become bot AI scripts and game VM logic.

## Design Patterns & Rationale

**Token-Based Processing**: Unlike traditional text-based preprocessors, this preprocessor operates on a pre-tokenized stream (`Tokenrow`), enabling precise macro detection and expansion without regex. A token's `type`, `len`, and `t` (text pointer) enable O(1) lookup of macro parameters and reliable `##` concatenation.

**Hideset Anti-Recursion**: Each token carries a `hideset` bitmap recording which macros have already expanded at that position. `checkhideset()` prevents infinite loops from self-referential macros (e.g., `#define X X`). This is copied to all tokens generated by expansion.

**Lazy Argument Evaluation**: `substargs()` does NOT pre-expand arguments unless they appear adjacent to `##` or as the `#` stringify operator target. This preserves lazy semantics and avoids spurious expansion of macro invocations inside argument values.

**Separate Built-In Macro Handling**: `builtin()` handles `__LINE__`, `__FILE__`, `__DATE__`, `__TIME__` via a switch on `biname` enum, each expanding to a synthetic token without requiring storage in the macro symbol table. This avoids polluting the definition table and allows per-invocation evaluation (e.g., line numbers).

**Two-Phase Concatenation**: `##` operators inside macro bodies are flagged as `DSHARP1` during argument substitution to suppress premature expansion, then reverted to `DSHARP` and processed in a final pass via `doconcat()`. This ensures correct left-to-right evaluation.

## Data Flow Through This File

1. **Definition Phase** (`dodefine()` or `doadefine()`):
   - Input: `Tokenrow` containing `#define NAME [ARGS] [BODY]` or command-line `-DNAME=VALUE`
   - Parse formal parameters into `args` tokenrow; normalize body into `def` tokenrow
   - Store in `Nlist` table under macro name with `ISDEFINED` flag
   - Output: Macro stored in symbol table, ready for expansion

2. **Lookup & Expansion** (`expandrow()` → `expand()`):
   - Input: Token stream with embedded NAME tokens
   - `expandrow()` scans for `NAME` tokens; `quicklook()` fast-path filters non-macro names
   - `lookup()` fetches `Nlist` entry; check `ISDEFINED` and `hideset` to determine if expansion occurs
   - Output: Expanded tokens inserted back into stream at macro name position

3. **Argument Gathering** (`gatherargs()`):
   - Input: Token stream at macro name; scans forward for `(` then gathers comma-separated argument groups
   - Handles nested parentheses, line continuations (newlines → spaces), and `EOF` robustness
   - Output: Array `atr[]` of `Tokenrow` pointers, one per argument; `*narg` = argument count

4. **Substitution** (`substargs()`):
   - Input: Macro replacement body `rtr`, argument array `atr[]`, macro definition `np`
   - Iterate over tokens; if `#` operator, stringify argument; if `##` adjacent, insert argument as-is; otherwise expand argument recursively
   - Output: Modified `rtr` with placeholders replaced by argument tokens

5. **Concatenation** (`doconcat()`):
   - Input: Macro body after argument substitution, with `DSHARP` and `DSHARP1` tokens
   - For each `DSHARP`, extract left and right token text, concatenate, re-tokenize via `gettokens()`, replace `ltp` and `ntp` with result
   - Output: Macro body with `##` operators resolved into new tokens

6. **Final Insertion**:
   - Copy macro body to new `Tokenrow`, distribute combined `hideset` to all NAME tokens, insert into token stream, rewind stream pointer for re-expansion of substituted arguments

## Learning Notes

**Token-Centric Preprocessing**: This file exemplifies how a token-based preprocessor differs from text-based ones (e.g., GNU CPP). By operating on an Abstract Syntax Tree-like token stream, it avoids fragile regex-based pattern matching and enables deterministic behavior for edge cases (e.g., `define f(x)f` when followed by `(y)` should NOT expand).

**Practical Macro Implementation Details**:
- Formal parameter matching is done by string comparison (`strncmp`), not by token identity, allowing renamed parameters
- `lookuparg()` returns the parameter index, enabling O(1) substitution
- Hideset is a bitmask (likely one bit per macro name up to ~32 macros), not a list, for speed and memory

**Idiomatic Q3/LCC Patterns**:
- Heavy reliance on global state (`cursource`, `outp`) rather than passing context through function arguments. This is typical of 1990s compiler infrastructure.
- Memory is managed via `dofree()` wrapper (likely a no-op or a pool-based allocator), reflecting LCC's emphasis on efficiency
- Built-in macros evaluated on-the-fly (fetching `curtime`, `cursource->line`, `cursource->filename`) rather than baked at compile-start time

**Comparison to Modern Approaches**:
- Modern preprocessors (e.g., in Clang) use a token buffer that is mutable in-place; this preprocessor allocates new `Tokenrow` objects for each expansion.
- No pragma support, no conditional compilation (`#if`, `#ifdef`—those are likely in `cpp.c`'s main loop).
- No module system or macro hygiene (Scheme-style `syntax-rules` or Rust `macro_rules!` patterns)—purely text substitution with hideset-based loop prevention.

## Potential Issues

1. **Buffer Overflow in `stringify()`** (line ~380): Fixed `STRLEN=512` buffer for stringified token content. If a macro argument contains many long tokens or escape sequences, the output could exceed 512 bytes. The check `sp+2*tp->len >= &s[STRLEN-10]` is approximate and could be optimistic if many tokens have escaping.

2. **Stack Depth in Recursive `expandrow()` Calls** (line ~314, via `substargs()` → `expandrow()` → `expand()`): No explicit recursion limit. A pathological macro definition could exhaust stack (e.g., `#define f(x) f(x)` triggered inside argument expansion). The `hideset` prevents *direct* recursion but not *mutual* recursion between macros.

3. **Hard-Coded Limits**:
   - `NARG` (likely ~32, line ~267) limits macro parameters; no runtime check for exceeding it beyond a `FATAL` error
   - `gatherargs()` assumes `gettokens()` will eventually produce an `END` token; if the file ends malformed, the `EOF in macro arglist` error may not gracefully recover

4. **Whitespace Loss in Argument Substitution**: The `XPWS` flag (line ~349) controls whether whitespace before a token is preserved. If a macro argument is substituted in a context where the preceding whitespace matters (e.g., after `#` operator), the result might be incorrect or ambiguous.
