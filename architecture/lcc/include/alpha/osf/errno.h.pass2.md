# lcc/include/alpha/osf/errno.h — Enhanced Analysis

## Architectural Role

This file is part of the **LCC compiler's platform-specific standard C library headers**, used when compiling Quake III QVM bytecode. The LCC toolchain maintains a directory of headers for different target platforms (Alpha/OSF, x86/Linux, x86/Win32, MIPS/IRIX, Sparc/Solaris); `errno.h` provides minimal error-code definitions needed for runtime error handling in compiled game/cgame/ui code. Though sparse compared to real OSF/1's errno.h, it supplies just enough to allow compiled QVM modules to reference standard error constants.

## Key Cross-References

### Incoming (who depends on this)
- **QVM compilation pipeline**: Game code (`code/game/*.c`), cgame VM (`code/cgame/*.c`), UI VM (`code/q3_ui/*.c`, `code/ui/*.c`) compiled with LCC will include this header when:
  - Standard library functions are used (e.g., `fopen`, `fread` which may reference `errno`)
  - Custom error-handling code checks EDOM (domain error) or ERANGE (range error)
- **LCC preprocessor**: Direct inclusion via `#include <errno.h>` in QVM source files

### Outgoing (what this depends on)
- No runtime dependencies; purely a compile-time interface
- Assumes the **LCC compiler itself** implements `errno` as an external int variable in its runtime or the target QVM runtime

## Design Patterns & Rationale

**Pattern: Minimal Standard Library**
- LCC provides only essential errno codes (EDOM=33, ERANGE=34), not the full POSIX set
- This reflects QVM's simplified sandbox: QVM code rarely needs filesystem/network error codes
- The `extern int errno` declaration allows compiled code to read/write the global error variable without linking against system libc

**Why this structure?**
- LCC must support multiple platforms with different errno layouts (Alpha/OSF uses 33/34; Linux/Win32 differ)
- Centralizing platform-specific headers in `lcc/include/<arch>/<os>/` lets the compiler swap headers per target without recompilation
- QVM code generated by this compiler is architecture-neutral (bytecode), but the **compilation environment** is platform-specific

**Tradeoff: Completeness vs. Footprint**
- Full OSF/1 errno.h would define 50+ error codes; LCC includes only what's used in practice
- QVM modules that need more error codes would have to define them locally or not use them

## Data Flow Through This File

1. **Compile-time:** Developer writes QVM C source → LCC preprocessor encounters `#include <errno.h>` → selects `lcc/include/alpha/osf/errno.h` based on `-m` architecture flag → defines EDOM/ERANGE macros and reserves space for extern errno
2. **Generated code:** LCC emits QVM bytecode with references to EDOM/ERANGE as constant immediates and errno as a data-segment symbol
3. **QVM execution:** At runtime, the game/cgame/ui VM (hosted by `qcommon/vm.c`) may read/write the errno variable via the VM's dataMask-confined data heap
4. **No actual error reporting:** Unlike native C, QVM modules rarely actually **set** errno; the `extern` declaration allows them to **check** it if engine syscalls or intrinsic functions report errors via errno

## Learning Notes

- **Idiomatic to this era/engine:** Quake III's QVM sandbox accepts pre-compiled bytecode without full system libc; hence minimal errno support. Modern engines (e.g., Godot, Unreal) use full language runtimes with complete stdlib, but Q3 trades completeness for portability and security.
- **Cross-platform header management:** LCC's `include/<arch>/<os>/` layout is a pre-autoconf pattern for handling platform diversity. Modern toolchains use `-isysroot` and `--target` flags; here, the headers themselves are duplicated per platform.
- **Connection to QVM security model:** The sandboxed QVM environment doesn't let bytecode call system functions directly (e.g., `open()`, `socket()`). Even if QVM code included full errno, those syscalls don't exist in the engine's trap vector. Thus EDOM/ERANGE suffice for math-library error codes.

## Potential Issues

- **None evident.** This is a minimal, correct header for QVM compilation on Alpha/OSF. If there were a scenario where QVM code needed additional errno codes, it would redefine them locally.
