# q3map/glfile.c — Enhanced Analysis

## Architectural Role

This file provides debug visualization output for the BSP portal structure during offline map compilation. It occupies a narrow, optional niche in the q3map tool pipeline: after portal generation but before final BSP writing, it can export the complete portal graph to a text-based format readable by external 3D visualization tools (GLView). Being a tool-only module, it has no runtime presence; it's purely a developer aid for inspecting and validating the spatial partitioning generated by the compiler.

## Design Patterns & Rationale

**Recursive tree traversal**: `WriteGLView_r()` mirrors the idiom used throughout bspc for spatial-tree processing—depth-first leaf iteration to enumerate portals. Avoids separate queue or stack data structure.

**Lightweight state visualization**: The `static int level` counter in `OutputWinding()` uses recursion depth to modulate RGB color, creating visual depth layering in 3D viewers without explicit depth values. Idiomatic 1990s—pragmatic rather than data-rich.

**Minimal formatting**: Plain ASCII output (not binary) favors tool interoperability and human inspection over compactness. Each portal becomes a human-readable winding with 6 floating-point values per vertex (3 position, 3 RGB).

**Portal visibility filtering**: `PortalVisibleSides()` applies opaqueness semantics from the CSG pipeline (stored in `node.opaque`) to avoid redundant or invisible portal faces—only emits geometrically meaningful surfaces.

## Data Flow Through This File

**Entry**: BSP tree post-processing phase passes a fully-built `tree_t` and map source name.
**Processing**: 
1. Recursive tree walk from `tree->headnode`, skipping non-leaf nodes
2. At each leaf, enumerate its portal list, filtering by `PortalVisibleSides()`
3. For visible portals, write winding vertex data (position + RGB)
4. Reverse winding if viewing from opposite side

**Exit**: Single `.gl` file (e.g., `maps/dm1.gl`) containing all portals in viewer-friendly format. Global `c_glfaces` counts total faces output.

## Learning Notes

- **Debugging archaeology**: This exemplifies pre-modern in-engine visualization. Modern engines use real-time debug overlays; Q3 outsourced portal debugging to standalone external tools.
- **Portal semantics**: The `PortalVisibleSides()` logic reveals that portals connect nodes with different content types (opaqueness). Two nodes with identical content share no visible boundary.
- **Iterative tree structure**: The linked-list portal chains per node (`node->portals`) show spatial partitioning was stored incrementally during BSP generation, not as a separate data structure.
- **Winding convention**: Portals maintain a front/back distinction; reversing signals to the viewer which side is being examined, consistent with Q3's surface handedness throughout the engine.

## Potential Issues

- **No error recovery**: File write failures (`fopen`, `fprintf`) are unchecked except at open time; partial/corrupt `.gl` output possible if disk fills mid-write.
- **Global mutable state**: The `static level` in `OutputWinding()` is non-reentrant; calling `WriteGLView()` twice in one process produces identical colors regardless of actual tree depth, though this is a non-issue in practice (tool is single-batch).
