# q3radiant/Brush.cpp — Enhanced Analysis

## Architectural Role

`Brush.cpp` is the core brush manipulation layer for the Q3Radiant level editor, implementing the in-memory representation and lifecycle of brush-based level geometry. As part of the editor toolchain (orthogonal to the runtime engine), it bridges the user's interactive brush construction with the BSP/AAS compilation pipeline: brushes here are CSG primitives that eventually feed into `q3map/brush.c` for BSP generation and `code/bspc/` for bot navigation mesh synthesis. The file manages the dual texture coordinate systems (standard vs. brush primitives mode), which is an idiosyncratic Q3 editor feature absent from modern engines.

## Design Patterns & Rationale

**Plugin-Aware Face Allocation**: `Face_Alloc()` conditionally calls `g_SurfaceTable.m_pfnTexdefAlloc()` if the surface-properties plugin is loaded. This reflects Radiant's extensible architecture—third-party plugins can attach custom texture definition data without modifying the core.

**Clone vs. FullClone dichotomy**: 
- `Face_Clone()` copies only `texdef` and plane points; derived fields (plane normal, winding) are regenerated by `Brush_Build()` (not shown).
- `Face_FullClone()` duplicates *everything* including winding and computed plane, used when an exact geometric copy is needed without recomputation. This shows thoughtful optimization: avoid redundant plane math where possible.

**Dual Texturing Modes**: The conditional branches in `Face_MoveTexture()` and `Face_TextureVectors()` handle two incompatible texture coordinate systems:
- **Standard**: Texture shifts are rotation-aware; scale factors directly modulate XYZ→ST mapping.
- **Brush primitives**: A Q3-specific, projection-agnostic system (details deferred to `Face_MoveTexture_BrushPrimit`, not shown). This fragmentation is historical—brush primitives were introduced post-Quake II to support arbitrary face orientations without projection artifacts.

**Texture Coordinate Clamping**: `Clamp()` wraps shifts modulo texture width/height—mimicking runtime behavior where shifts beyond image bounds wrap. Editor and engine must agree on this wrapping to avoid surprises.

## Data Flow Through This File

1. **Creation**: `Brush_Alloc()` → (via caller) `Face_Alloc()` × N faces.
2. **Geometry setup**: Caller sets `face_t.planepts[3]` (three points defining the face plane).
3. **Plane derivation**: `Face_MakePlane()` computes `plane_t` (normal + distance) from the three points via cross product; no face winding yet.
4. **Texture assignment**: `f->d_texture = Texture_ForName(...)` looks up texture image; `texdef_t` holds shift, scale, rotation.
5. **Rendering prep**: `Face_SetColor()` applies plane-based shading (via `SetShadeForPlane`) and texture color sampling for viewport display.
6. **Export**: When saving, the brush's faces (with planes, texdef, winding) are serialized to `.map` format; q3map later rebuilds windings from planes for BSP compilation.

**Key asymmetry**: Editor stores **planes + texture metadata**; windings are derived on-demand. Runtime BSP stores **windings only** (see `q3map/brush.c` for comparison).

## Learning Notes

**Idiomatic to Radiant/Q3**: 
- The explicit **baseaxis** lookup table (18 base vectors for 6 cardinal plane orientations) is a classic "compress texture-axis selection into a lookup" pattern, avoiding conditionals in the inner loop. Modern engines use matrix rotations or direct UV parameterization.
- **Brush primitive mode** is purely editorial—a way to author complex geometry without suffering from axial-plane texture-coordinate singularities. The runtime engine doesn't know or care; the mode only affects how Radiant exports `.map` files to q3map.

**Cross-Subsystem Insight**: 
- This file's `Face` type differs from `srfTriangle_t` / `drawSurf_t` in the renderer (`code/renderer/tr_*.c`). The editor works with abstract CSG geometry; the renderer works with already-tesselated, shader-assigned triangles. The pipeline is **one-way**: editor → map → bsp compiler → renderer.

**Plugin System Coupling**: 
The conditional `g_qeglobals.bSurfacePropertiesPlugin` calls show Radiant's plugin hooks were wired at allocation time, not via virtual dispatch. A plugin presence is checked *once* per Face lifecycle, then the plugin's function pointers are cached. This is safe because plugins are statically loaded at startup.

## Potential Issues

1. **Brush Primitive Mode Divergence**: `Face_MoveTexture()` has separate code paths that must remain synchronized. If a developer fixes a bug in standard-mode texture shifting but forgets brush-primitive mode (or vice versa), subtle map corruption occurs silently. The code has a commented-out debug warning (`#ifdef _DEBUG` → `Sys_Printf`), but no automated tests.

2. **Plugin Lifecycle Brittle**: If `g_SurfaceTable.m_pfnTexdefAlloc()` fails or the plugin unloads mid-edit, `f->pData` becomes dangling. The destructor in `Face_Free()` calls `GETPLUGINTEXDEF(f)->DecRef()` unconditionally if `pData` is non-null, trusting the plugin's ref-counting. A mismatch (alloc in plugin A, free with plugin B) could crash.

3. **Texture Lookup Without Fallback**: `Texture_ForName()` is called in `Face_FullClone()` and `Face_SetColor()` without null checks. If the texture name is invalid or the texture system hasn't initialized, `f->d_texture` becomes null, and downstream `f->d_texture->width/height` dereferences crash.

4. **No Winding Validation Post-Plane**: `Face_MakePlane()` computes a plane from three points but doesn't validate that the winding (when later computed) lies on that plane. Numerical errors in point selection can lead to degenerate or reversed windings, discovered only when q3map reads the exported map.
