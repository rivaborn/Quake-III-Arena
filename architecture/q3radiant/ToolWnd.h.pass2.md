# q3radiant/ToolWnd.h — Enhanced Analysis

## Architectural Role

`ToolWnd.h` defines a minimal MFC window base class for Radiant's tool windows. It bridges the Microsoft Foundation Classes framework (used throughout q3radiant for Win32 UI) with the editor's custom tool panel infrastructure. This file is entirely orthogonal to the runtime engine architecture: Radiant is an offline level editor with zero coupling to qcommon, renderer, or any game subsystems. All tool windows (entity properties, shader browsers, terrain editors, etc.) likely inherit from `CToolWnd` to participate in Radiant's window docking and layout management.

## Key Cross-References

### Incoming (who depends on this file)
- Other q3radiant tool window classes likely inherit from `CToolWnd` (e.g., shader editors, entity dialogs, scripting windows)
- `MainFrm` (main editor frame) probably manages a collection of `CToolWnd`-derived windows and their visibility/docking state
- The `PlugIn` system (`PlugIn.h`, `PlugInManager.h`) may expose a similar base for third-party tool windows

### Outgoing (what this file depends on)
- **MFC framework only**: `CWnd` (Windows base window class from `afxwin.h`, not included here but implicitly required)
- No engine dependencies whatsoever (no qcommon, renderer, physics, etc.)
- No cross-compilation to VMs or platform layers

## Design Patterns & Rationale

**MFC ClassWizard Template Pattern**: The file is essentially a stub generated by Visual Studio's ClassWizard tool. The `//{{AFX_VIRTUAL`, `//}}AFX_VIRTUAL` and `//{{AFX_MSG}}` markers are ClassWizard directives—they signal where the wizard injects virtual overrides and message map entries. This is a hallmark of late-1990s/early-2000s Visual C++ development.

**Intentional Emptiness**: The lack of any custom data members or methods is intentional. `CToolWnd` is a pure base class; each subclass brings its own content (buttons, text fields, trees, etc.). This design mirrors the composite pattern seen in game engine architecture, but applied to UI framework objects instead of game entities.

**Message Map Abstraction**: The `DECLARE_MESSAGE_MAP()` macro registers this class with MFC's message-dispatch system, allowing Windows messages (WM_PAINT, WM_LBUTTONDOWN, etc.) to be dispatched to virtual overrides declared by subclasses. This is entirely alien to the game engine's frame-based event model (which uses fixed entity ticks and syscalls).

## Data Flow Through This File

No data flows through this file. It is a pure template/structural class:
- **Input**: None. Windows messages are dispatched *to* subclasses, not *through* this base.
- **Output**: None. The window renders via inherited `CWnd::OnPaint` or subclass overrides.
- **State**: None. No members, no static globals.

## Learning Notes

**What a Developer Studying This Engine Learns**:
- Radiant is a **standalone Win32 application**, not a runtime DLL or platform layer
- The editor uses MFC (Microsoft Foundation Classes), a pre-template era framework for rapid UI development
- Tool window infrastructure is decoupled from game logic entirely—there is no hot-reload, no VM linkage, no physics integration into the editor UI
- The sparse design reflects the era: minimal base classes, explicit message dispatch, no modern event-binding frameworks

**Era-Specific Patterns**:
- MFC and ClassWizard were the dominant Windows UI framework in the early 2000s (before .NET/WPF)
- The `//{{` comments are MFC-era metadata that allowed IDEs to preserve hand-written code while regenerating boilerplate
- Compare to modern game editors (Unity, Unreal): they embed full game engines in the editor for real-time preview and use language-native event systems, not message maps

**Connection to Engine Concepts**:
- This is **not** an ECS, **not** a subsystem, **not** a VM module
- It's pure UI framework plumbing—worlds away from the entity/component model of the game engine or the shader/surface/surface system of the renderer

## Potential Issues

None inferable from this file alone. It's a valid MFC base class. The entire q3radiant subsystem is orthogonal to runtime code, so bugs here cannot propagate to gameplay, network, or rendering pipelines. The primary risk is **Windows XP/Vista forward-compatibility**: MFC is deprecated on modern Windows, and this code would struggle on x64 builds or Windows 10+.
